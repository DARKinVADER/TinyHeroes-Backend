# backend/CLAUDE.md

## Clean Architecture

Four projects with strict dependency direction (`Api` → `Application` ← `Infrastructure`, both → `Domain`):

- **Domain** (`TinyHeroes.Domain`) — entities and enums only, no dependencies
- **Application** (`TinyHeroes.Application`) — DTOs, service interfaces (`ITokenService`, `ISummaryService`, `IAiImageService`, `IFileStorageService`), custom exceptions (`NotFoundException`, `ForbiddenException`)
- **Infrastructure** (`TinyHeroes.Infrastructure`) — `AppDbContext` (EF Core + ASP.NET Identity), service implementations, `DependencyInjection.cs` extension method
- **Api** (`TinyHeroes.Api`) — controllers, `ExceptionHandlingMiddleware` (maps `NotFoundException` → 404, `ForbiddenException` → 403), `Program.cs`

`Program.cs` runs `Database.Migrate()` on startup for relational databases; migrations must be created manually with `dotnet ef migrations add` when the schema changes. Tests use EF InMemory, so no Postgres is needed to run them.

### Dependency rules

Violations to reject:
- No EF Core (`DbContext`, `DbSet`, LINQ to entities) in `Application` or `Domain`
- No ASP.NET types (`HttpContext`, `IActionResult`) outside `Api`
- No `Infrastructure` namespace references from `Api` controllers — use `Application` interfaces only
- No business logic in controllers — controllers orchestrate, services execute

When adding a new capability: interface in `Application/Interfaces/`, implementation in `Infrastructure/Services/`, registered in `DependencyInjection.cs`.

New prize or deed types extend via data (presets table), not new controller branches or `if/switch` on type.

## Authorization Pattern

All controllers extend `ApiControllerBase`, which exposes `GetUserId()` to extract the authenticated user's ID from the JWT. Family-scoped endpoints verify membership by querying `FamilyMembers` for a record matching both `UserId` and `FamilyId` before touching any family-owned data. Missing → `NotFoundException` (→ 404); present but wrong role → `ForbiddenException` (→ 403).

## Domain Model

Key relationships:
- `FamilyMember` — pivot between `User` and `Family`; carries the member's role
- `GoodDeed` — belongs to `Child` (not directly to `Family`)
- `PrizeAssignment` — weekly/monthly winner record; `PrizeClaim` is the parent confirming delivery
- `WeekSummary` / `MonthSummary` — pre-computed ranking snapshots generated by `SummaryService`
- `Family.WeeklyMinDeeds` / `Family.MonthlyMinDeeds` — optional thresholds; children must reach these to be prize-eligible

## Auth Flow

The API uses JWT as the primary auth scheme. Social login (Google, Apple, Facebook) is configured conditionally based on whether `Auth:*:ClientId` is non-empty in config.

Social login is a two-step redirect: the provider calls `/api/auth/social/{provider}/callback`, which issues a short-lived code and redirects the browser to the Angular route `/auth/callback?code=<code>`. `CallbackComponent` then exchanges the code via `POST /api/auth/exchange` to receive the JWT. This avoids putting the token in the URL directly.

## Tests

Each `TestWebApplicationFactory<Program>` instance creates a uniquely-named in-memory database (`TestDb_<Guid>`), so test classes are fully isolated — no shared state between files. It also swaps `IAiImageService` (returns fake base64) and `IFileStorageService` (returns a fake path).

`TestAuthHelper` provides:
- `RegisterWithFamily(factory)` — registers a user, creates a family, returns an authorized `HttpClient`
- `RegisterOnly(factory)` — registers a user only (for testing the no-family state)

Every new feature ships with integration tests in `TinyHeroes.Tests/Integration/`. Do not add new test setup patterns. Cover:
- Happy path (correct input, expected HTTP status + response body)
- Auth boundary (unauthenticated → 401, wrong family → 404, wrong role → 403)
- Not-found case where applicable

New services with non-trivial logic get unit tests in `TinyHeroes.Tests/Unit/`.
